为什么不应该通过尺寸来判断可触摸设备

发表于 2017-03-20 | 分类于 前端技术 ， CSS |

原文地址 Touch Devices Should Not Be Judged By Their Size

前端工程师和网页设计师全都生活在多媒体设备中。

几个月前，红帽子UXD团队讨论如何设计企业应用程序在移动端环境上。我的两位非常有才华的同事,Sarah和Jenn,指出在移动端不能只依靠尺寸来判断触摸设备。

设想是容易的，如果我们都能达成一致，那么网页设计也不会那么难。

  • Jeremy Keith, Resilient Web Design

今天，对于已经复杂的交互设计和前端开发来说，又多了一项新的挑战。

硬件行业创造了大量的触摸屏电视，真正的大型平板电脑（如iPad Pro），甚至是巨大的触摸式台式电脑（如新的不太好用的Surface Studio）。 这意味着我们不能再假设一个小视口是触摸屏，而且没有大的视口。 有时大屏幕触摸，要求用户使用他们的手指，小屏幕有触控笔。

响应式的视口媒体查询很棒，但还不够。

我们可以使用像Modernizr这样的JS工具来检测触摸屏，但CSS具有更加智能，更加神奇的功能。

###互动媒体功能

感谢W3C CSS工作组和CSS社区，我们有一个更简洁的解决方案。

在媒体查询4级工作草案中，有一个交互媒体功能的规范，包括三个定义：

  • 指点设备质量
  • 悬停能力
  • 罕见的互动能力

这些提供了基于用户的指点设备的存在和准确性以及是否具有将数据悬停在元素上的能力来查询文档的能力。

我们来仔细看看每一个：

指点设备质量：指针特性

指针媒体功能用于查询诸如鼠标之类的指示设备的存在和准确性。 如果设备具有多种输入机制，则指针媒体功能必须反映由用户代理确定的“主”输入机制的特征。“ - W3C

这里的关键词是指向设备的“准确性”。

  • 鼠标或绘图手写笔非常准确，值为fine。

  • 手指或Kinect外围设备不是，值为coarse。

因此，我们可以将UI元素适应用户的指针功能。 如果用户的主输入机制是手指，这对于使命中区域更大是有用的。

这个示例用来自适应点击单选或复选的区域尺寸。

###悬停能力：悬停功能

悬停媒体功能用于查询用户将鼠标悬停在页面上的元素上。 如果设备具有多种输入机制，则悬停媒体功能必须反映用户代理确定的“主”输入机制的特征。“ - W3C

需要重点注意的是，它只对主要的输入机制进行判断。 如果主输入机制无法悬停，但辅助输入可以，则查询将解析为none：

例如，触摸屏长按被视为hover：none。“ - W3C

  • 触摸屏设备，其中主指针系统是手指，不能悬停，将取值为none。

  • 主输入是鼠标并且可以轻松地悬停页面部分的设备取值为hover。

充分利用这个查询是一个下拉菜单。

###不常用的交互功能：任何指针和任何悬停

在触摸并具有鼠标或触控笔（如Microsoft Surface）的设备上，悬停和指针媒体查询仅作为评估输入的主要机制。

正如Andrea Giammarc指出的，他的戴尔XPS 13“触摸值是fine，虽然它有一个触摸屏，主要因为输入机制是一个鼠标。

@Real_CSS_Tricks FYI我有一个触摸屏，该方法就像Chrome中的魔法一样失效。 😕

  • Andrea Giammarchi（@WebReflection）2017年2月5日

如果我们想要一个这样的设备采取coarse或hover的值，我们可以使用少有的交互能力。

任何指针和悬浮的媒体功能与指针和悬停媒体功能相同，但它们对应于用户可用的所有指点设备的功能的并集。 如果不同的指点设备具有不同的特征，它们中的一个以上的值可以匹配。 如果所有的指针设备都不匹配相应的查询，或者根本没有指向设备，那么它们只匹配空。“ -W3C

###设备的例子

匹配指针和悬停组合的设备的典型示例：

pointer: coarse pointer: fine
hover: none smartphones, touch screens stylus-based screens (Cintiq, Wacom, etc)
hover: hover Nintendo Wii controller, Kinect mouse, touch pad

W3C

Patrick H. Lauke为每个交互设备类型写了一个很好的媒体查询指南。

这真的很酷，对吧？我听到你说：浏览器支持怎么样？

###浏览器并不是全都不支持！

尽管这是一个工作草案，但它仍有很好的支持。

我的简单测试在Chrome，Chrome for Android，Safari，Edge，Opera，Samsung浏览器和Android浏览器上都取得了成功，但在FireFox，Opera Mini或IE上无效。

FireFox和IE仅占移动/平板浏览器市场份额的2％以上。 我找不到有关触摸电视或其他不是手机或平板电脑的触摸屏设备的信息。

我想我们已经准备好使用这个功能了，而且随着FireFox增加了对它的支持，IE也不再维护旧版本，我们将得到全力的支持。

###“卡片选择”用例

一个月前，我们致力于为新版本的PatternFly实现一个多选卡组件，由我提供的开源设计系统。 这是使用悬停和指针媒体查询的完美案例。

要选择一张卡片，当用户将鼠标悬停在其上时，会显示一个复选框。 如果用户无法将鼠标悬停在元素上，那么我们随时会显示该复选框。

为了改善这种交互，如果主输入机制很粗糙，我们增加了复选框的命中面积。

Firefox和IE将始终显示默认复选框。

###尺寸不是一切

设备应该由他们的能力来判断，因为最终是定义它们的是功能。

这是一个未充分利用的功能，它打开了令人兴奋的新挑战的大门。 我迫不及待想看到我们作为一个社区，可以做什么。
